# Position-Aware Formation Placement — Design

**Date:** 2026-06-27

## Problem

The team-view "Formation" bento card shows a pre-baked PNG
(`assets/pitches/<slug>-<theme>.png`) generated by `wc2026_pitches.py`. The
generator places players by blindly zipping mplsoccer's formation slots
(correctly ordered GK→def→mid→att) against the `xi` array in
`assets/data/estimated_starting_eleven.json` **in array order, with no
knowledge of each player's real position**.

When the JSON's player order does not cleanly split into the formation's
lines, players land in the wrong slot. Example — **Argentina (4-3-3)**: Alexis
Mac Allister (a Central Midfielder) is placed on the **Right Wing**, and Nico
González (a winger) is placed in central midfield. The formation digits are
correct; the player→slot mapping is wrong.

## Goal

Place each starter on the slot that matches their **real position**, keeping
the formation digits and the existing static-PNG architecture. Fix the root
cause (no position awareness), not the symptom.

## Data available

- `assets/data/squads.csv` has every player's detailed position
  (`Goalkeeper`, `Centre-Back`, `Left-Back`, `Right-Back`,
  `Defensive Midfield`, `Central Midfield`, `Attacking Midfield`,
  `Left Midfield`, `Right Midfield`, `Left Winger`, `Right Winger`,
  `Centre-Forward`, `Second Striker`).
- It joins to the estimated XI by `(canonical team, shirt_number)`. Verified:
  **0 missing players** across all 48 teams.
- `scipy` and `mplsoccer` are already present at build time.

## Mechanism — position-aware optimal assignment

New pure module **`src/data/formation_assign.py`** (no mplsoccer/scipy
leakage into the running app; the renderer is the only caller that touches
mplsoccer):

- `NATURAL_POSITIONS: dict[str, tuple[float, float]]` — each raw squads.csv
  position → a natural spot on the opta pitch, oriented to match mplsoccer
  (x = depth, 12 GK → 88 striker; y = width, 12 = right … 88 = left):

  | Position | (x, y) | | Position | (x, y) |
  |---|---|---|---|---|
  | Goalkeeper | (12, 50) | | Attacking Midfield | (72, 50) |
  | Right-Back | (27, 14) | | Right Winger | (76, 12) |
  | Centre-Back | (27, 50) | | Left Winger | (76, 88) |
  | Left-Back | (27, 86) | | Second Striker | (84, 50) |
  | Defensive Midfield | (44, 50) | | Centre-Forward | (88, 50) |
  | Right Midfield | (58, 14) | | | |
  | Central Midfield | (58, 50) | | | |
  | Left Midfield | (58, 86) | | | |

- `natural_xy(position) -> (x, y)` — lookup with neutral centre fallback
  `(58, 50)` for unknown/missing positions.

- `assign_xi_to_slots(slots, xi, positions) -> list` — the core.
  - `slots`: list of `(x, y)` slot coordinates, in mplsoccer order (the
    renderer passes `pitch.get_formation(formation)` coords). `slots[0]` is
    always the GK slot.
  - `xi`: the 11 `(surname, number)` tuples.
  - `positions`: `{shirt_number: raw_position}` for the team.
  - **Pins the goalkeeper**: the single player whose `position ==
    "Goalkeeper"` is assigned to `slots[0]`; if not exactly one goalkeeper is
    found, fall back to including everyone in the optimisation.
  - Runs `scipy.optimize.linear_sum_assignment` over the remaining players ×
    slots, minimising squared Euclidean distance between each player's natural
    spot and each slot.
  - Returns `xi` **reordered to align with `slots`** (result[i] is the player
    for slots[i]). The renderer's existing `zip(slots, xi)` is untouched.

### Why optimal assignment

When a line's real-position count ≠ the formation digit (e.g. 4 central
players competing for 3 central slots), global optimal assignment slides the
most advanced player into the nearest free attacking slot, instead of dumping
a midfielder on the wing. Guarantees a bijection: every slot filled once,
every player placed once.

## Renderer change — `wc2026_pitches.py`

1. Build `positions` for the team via the existing `SquadRepository`
   (`src/data/squads.py`), keyed by shirt number.
2. Get slot coordinates from mplsoccer: `[(p.x, p.y) for p in
   pitch.get_formation(formation)]`.
3. `assigned = assign_xi_to_slots(slots, xi, positions)`.
4. Render exactly as today, zipping slots with `assigned`.
5. Regenerate all 96 PNGs (`python wc2026_pitches.py`).

No app-runtime change. No new dependencies.

## Testing (TDD, pure module — no mplsoccer needed)

mplsoccer slot coordinates are hard-coded as literals in the tests (captured
from `pitch.get_formation`). Cases:

1. `natural_xy` maps known positions to the expected zones; unknown →
   `(58, 50)` fallback.
2. GK is pinned to `slots[0]`.
3. **Argentina 4-3-3 regression** — with Argentina's real positions, Mac
   Allister (Central Midfield) lands in a midfield slot (x ≤ 58), NOT the RW
   slot `(73, 10)`; the wingers land in the wide attacking slots.
4. Count-mismatch overflow: 4 central-ish players, 3 central slots → the
   extra goes to the nearest attacking slot; all slots filled.
5. Bijection: every player assigned exactly once, every slot filled once.
